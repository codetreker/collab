# Collab Server Go 重写 — PRD

日期：2026-04-25 | 状态：Draft

## 背景

Collab 是一个 Human-Agent 协作聊天平台，当前 server 基于 TypeScript/Node.js (Fastify)。
趁现在功能边界清晰、集成测试覆盖完善，决定用 Go 重写 server 端，
以获得更小的部署体积、更快的启动速度和更简单的运维。

## 目标用户

- **终端用户（人类 + Agent）**：使用 Collab 聊天、协作的所有用户——他们不应感知到 server 切换
- **Plugin 开发者**：通过 WS 连接 server 的 plugin——接口保持不变
- **运维 / 部署者**：受益于更小镜像、更快启动、单二进制部署

## 核心需求

### 需求 1：功能完全兼容

**用户故事**：作为 Collab 用户，我希望 server 切换后所有功能照常运行，无需任何操作。

**范围**：Go server 必须 1:1 覆盖现有 TS server 的全部功能，包括但不限于：

- **REST API 端点**（路径、请求格式、响应格式完全一致）：
  - 认证（注册、登录、Token 刷新、API Key）
  - 频道管理（CRUD、成员管理、排序、频道组）
  - 消息（发送、编辑、删除、分页、搜索）
  - 私聊 (DM)
  - 投票 (Poll)
  - 表情回应 (Reactions)
  - 文件上传与访问
  - 用户管理（个人资料、在线状态）
  - Agent 管理
  - 管理后台 (Admin)
  - 工作区 (Workspace)
  - Slash Commands（自定义命令注册与执行）
  - Remote Explorer（远程文件浏览）

- **WebSocket 连接**：
  - 客户端 WS（实时消息推送、在线状态、心跳）
  - Plugin WS（插件通信通道）
  - Remote WS（远程节点连接）
  - SSE Stream（消息流推送）

- **数据库兼容**：
  - 使用现有 SQLite 数据库文件，schema 完全兼容
  - 现有数据无需迁移即可被 Go server 读写

- **静态文件服务**：
  - Go server 直接 serve 前端构建产物
  - serve 上传文件目录

**验收标准**：

- [ ] 前端代码零修改，连接 Go server 后所有页面和功能正常
- [ ] 所有 REST API 端点的请求/响应格式与 TS server 一致
- [ ] 所有 WS 消息类型和协议与 TS server 一致
- [ ] Plugin WS 连接和通信正常
- [ ] Remote 节点连接和文件浏览正常
- [ ] 现有 SQLite 数据库可被 Go server 直接使用，数据完整

### 需求 2：部署简化

**用户故事**：作为运维人员，我希望部署更简单、镜像更小、启动更快。

**验收标准**：

- [ ] 单个 Go 二进制 + 前端 static files 即可运行完整服务
- [ ] Docker 镜像体积显著小于当前 Node.js 版本
- [ ] 冷启动时间显著优于当前 Node.js 版本
- [ ] 支持与现有部署相同的环境变量配置（PORT、HOST、CORS_ORIGIN、UPLOAD_DIR 等）

### 需求 3：测试覆盖

**用户故事**：作为团队成员，我希望 Go server 有充分的测试保障，确保功能正确和后续维护信心。

**验收标准**：

- [ ] 现有 TS server 的所有集成测试场景在 Go server 中有对应覆盖
- [ ] 测试覆盖率不低于 85%
- [ ] 关键路径（认证、消息收发、WS 连接、文件上传）有端到端测试

### 需求 4：平滑切换

**用户故事**：作为团队，我希望切换过程可控、可回滚。

**验收标准**：

- [ ] 现有 TS server 代码保留，切换期间可随时回退
- [ ] 提供明确的切换步骤文档
- [ ] 切换过程中用户数据不丢失

## 不在 v1 范围

- ❌ Client（前端）代码修改
- ❌ Plugin / Remote Agent 代码修改
- ❌ 新功能（先 1:1 对等，功能增强另开任务）
- ❌ 数据库 schema 变更（保持兼容）
- ❌ 多数据库支持（继续用 SQLite）

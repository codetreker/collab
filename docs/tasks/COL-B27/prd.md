# COL-B27 Admin 控制面板拆分 — PRD

日期：2026-04-26 | 状态：Draft

## 背景

当前系统存在以下问题：

1. **安全漏洞（P0）**：普通成员可以获取系统中所有用户的完整列表，包含管理员信息——这是严重的信息泄露
2. **权限边界模糊**：管理员专属功能和普通用户功能没有清晰隔离，容易出现越权访问
3. **入口混杂**：管理后台（用户管理、频道管理、设置等）和用户聊天界面混在一起，没有独立入口
4. **依赖不合理**：@mention 和私信功能依赖全量用户列表才能工作，权限修复后需要适配

## 目标用户

- **Admin（管理员）**：管理用户、频道、邀请码、权限，有全量数据视图
- **Member（普通成员）**：只能看到和自己相关的数据（同频道成员、可私信的人），不应接触任何管理能力

## 核心需求

### 需求 1: 管理员功能权限隔离

**用户故事：** 作为系统管理员，我希望管理功能有独立的权限边界，以便普通成员无法访问任何管理员专属功能。

**验收标准：**

- [ ] 所有管理员专属功能（用户管理、频道管理、邀请码管理、权限管理等）有统一的权限校验，非管理员一律无法访问
- [ ] 普通成员无法通过任何途径获取系统全量用户列表
- [ ] 所有原有管理功能在权限隔离后仍正常工作

### 需求 2: 管理后台独立入口

**用户故事：** 作为管理员，我希望管理后台与用户聊天界面分开，有独立入口，以便管理操作不干扰日常使用。

**验收标准：**

- [ ] 管理后台（用户管理、频道管理、邀请码管理、设置）有独立入口，与聊天页面分离
- [ ] 非管理员用户看不到管理后台的入口
- [ ] 管理员创建用户时不能选择 agent 类型（agent 由用户自己创建，关联 COL-BUG-022）
- [ ] 管理员用户列表不显示 agent（agent 有独立管理页面）

### 需求 3: @mention 和私信功能适配

**用户故事：** 作为普通成员，我希望 @mention 和私信功能在不暴露全部用户列表的前提下正常工作，以便我可以在权限范围内正常沟通。

**验收标准：**

- [ ] @mention 选人器基于当前频道的成员列表工作，不再依赖全量用户列表
- [ ] 私信列表基于已有的私信会话展示，不再依赖全量用户列表
- [ ] 创建新私信可以从频道成员或已有私信列表中选人
- [ ] 功能体验与修改前一致，用户无感知

## 不在 v1 范围

- WebSocket 连接的权限拆分
- 细粒度 RBAC 权限模型重构（当前只有 admin/member/agent 三种角色）
- 管理后台作为独立部署的服务（本次仅前端入口分离，后端仍是同一个服务）
- Agent 管理接口的权限变更（Agent 属于用户自己管理）
- 审计日志（管理员操作记录）

## 开放问题

1. **全量用户列表是直接移除还是保留给管理员？** 建议移除独立接口——管理员通过管理后台查看，普通成员通过频道成员列表获取所需信息
2. **在线用户列表是否需要限制？** 当前返回在线用户 ID（不含详细信息），安全风险较低，暂不处理
3. **频道成员信息是否足够支撑 @mention 选人器？** 需确认返回信息包含显示名、头像等必要字段
4. **管理后台入口形式？** 建议前端路由分离，降低改动成本
5. **Go server 和 TS server 是否都需要同步修改？** Go 重写（COL-R01）进行中，需确认

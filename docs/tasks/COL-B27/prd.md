# COL-B27 Admin 控制面板拆分 — PRD

日期：2026-04-26 | 状态：Draft

## 背景

当前系统存在以下问题：

1. **安全漏洞（P0）**：普通成员可以获取系统中所有用户的完整列表，包含管理员信息——这是严重的信息泄露
2. **权限边界模糊**：管理员专属功能和普通用户功能没有清晰隔离，容易出现越权访问
3. **入口混杂**：管理后台（用户管理、频道管理、设置等）和用户聊天界面混在一起，没有独立入口
4. **身份混淆**：管理员和普通用户使用相同的身份体系，导致 admin 出现在用户列表、可被 @mention 等不合理行为

## 目标用户

- **Admin（系统管理员）**：固定账号 admin，只能登录管理后台，管理用户、频道、系统设置。admin 是独立身份，不是普通用户
- **User（普通用户）**：通过邀请注册或管理员创建，使用聊天功能，管理自己的 agent
- **Agent**：由用户创建和管理的 AI 助手，归属于创建它的用户

## 核心需求

### 需求 1: 管理员身份独立

**用户故事：** 作为系统管理员，我的账号是独立于普通用户的身份，只能登录管理后台，不参与聊天，以便管理职能与用户行为完全隔离。

**关键决策：**

- 系统管理员账号固定为 admin，密码由部署时配置
- admin 是独立的身份，不是用户的一种角色
- admin 只能登录管理后台，不能作为普通用户聊天
- admin 不出现在用户列表中

**验收标准：**

- [ ] admin 账号只能登录管理后台，无法进入聊天界面
- [ ] admin 无法发送消息、加入频道
- [ ] admin 不出现在任何用户列表中（包括 @mention 选人器、私信列表、频道成员列表）
- [ ] 普通用户无法访问管理后台

### 需求 2: Admin 后台功能

**用户故事：** 作为管理员，我需要一个独立的管理后台来管理用户、频道和系统设置，以便集中处理所有管理操作。

**功能范围：**

- **用户管理**：查看用户列表、创建用户、管理用户状态
  - 只能创建 user，不能创建 admin 或 agent
  - **User Detail 页面**：查看用户完整信息，以及该用户创建的所有 agent（线框图见 [Admin 管理后台 - User Detail](../../ui/admin.md#4-user-detail)）
- **频道管理**：维持现有功能
- **系统设置**：维持现有功能

**安全红线：**

- admin 接口不返回任何用户的 API key（不只是前端不显示，后端接口也不返回）

**验收标准：**

- [ ] 管理后台有独立入口，与聊天页面分离
- [ ] 用户管理只能创建 user 类型账号
- [ ] User Detail 页面展示用户完整信息和该用户创建的 agent 列表
- [ ] 管理后台任何页面都不展示用户的 API key
- [ ] admin 相关 API 接口不返回 API key 字段
- [ ] 频道管理、系统设置功能正常

### 需求 3: 用户注册方式

**用户故事：** 作为新用户，我可以通过邀请链接或邀请码注册；作为管理员，我可以在后台直接创建用户账号。

**支持的注册方式：**

- 邀请注册：通过邀请链接或邀请码自助注册
- 管理员创建：admin 在管理后台直接创建用户

**验收标准：**

- [ ] 新用户可以通过有效的邀请链接/码完成注册
- [ ] 管理员可以在后台创建新用户
- [ ] 无有效邀请或管理员创建以外的途径无法注册

### 需求 4: Agent 管理归属

**用户故事：** 作为普通用户，我可以自己创建和管理 agent；管理员只能查看但不能替我操作。

**关键决策：**

- Agent 由用户自己创建和管理
- Admin 后台只能查看用户的 agent 列表（在 User Detail 页面内），不能替用户创建、编辑或删除 agent

**验收标准：**

- [ ] 用户可以创建、编辑、删除自己的 agent
- [ ] 管理后台 User Detail 页面可查看该用户的 agent 列表
- [ ] 管理员无法在后台替用户创建、编辑或删除 agent

### 需求 5: 前端 @mention 和私信适配

**用户故事：** 作为普通用户，我希望 @mention 和私信功能在不暴露全部用户列表的前提下正常工作，以便我可以在权限范围内正常沟通。

**验收标准：**

- [ ] @mention 选人器基于当前频道的成员列表工作，不再依赖全量用户列表
- [ ] 私信列表基于已有的私信会话展示，不再依赖全量用户列表
- [ ] 创建新私信可以从频道成员或已有私信列表中选人
- [ ] 功能体验与修改前一致，用户无感知

## 数据迁移

现有数据可以清空重建，不需要做数据迁移。

## 不在 v1 范围

- WebSocket 连接的权限拆分
- 细粒度 RBAC 权限模型重构（当前只有 admin/user/agent 三种身份）
- 管理后台作为独立部署的服务（本次仅前端入口分离，后端仍是同一个服务）
- 审计日志（管理员操作记录）
- 开放注册（无邀请的自助注册）

## 开放问题

1. ~~全量用户列表是直接移除还是保留给管理员？~~ **已决策**：管理员通过管理后台查看，普通用户通过频道成员列表获取所需信息
2. **在线用户列表是否需要限制？** 当前返回在线用户 ID（不含详细信息），安全风险较低，暂不处理
3. ~~频道成员信息是否足够支撑 @mention 选人器？~~ 需确认返回信息包含显示名、头像等必要字段
4. ~~管理后台入口形式？~~ **已决策**：独立入口，与聊天界面分离
5. **邀请码/链接的有效期和使用次数限制？** 待确认
